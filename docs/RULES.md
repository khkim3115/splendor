# 스플랜더(Splendor) 공식 룰 문서 — 웹 게임 구현 기준(Source of Truth)

> **검증 기준**: 이 문서는 Space Cowboys 발행 스플랜더 공식 영문 룰북(2014, Marc André 작)의 원문 전체를 확인하여 작성했다. 룰북 원문에 명시된 규칙은 "[룰북]", 공식 FAQ·통용 해석으로 보강한 규칙은 "[FAQ]", 룰북·FAQ 모두 침묵하여 구현 결정이 필요한 부분은 "[구현 결정]"으로 표기한다. 구현자는 이 문서만으로 룰 엔진을 작성할 수 있어야 하며, 표기가 [구현 결정]인 항목은 본 문서의 권장안을 기본값으로 삼는다.

## 1. 구성물 [룰북]

| 구성물 | 수량 | 비고 |
|---|---|---|
| 에메랄드(초록) 토큰 | 7개 | 보석 토큰 |
| 다이아몬드(하양) 토큰 | 7개 | 보석 토큰 |
| 사파이어(파랑) 토큰 | 7개 | 보석 토큰 |
| 오닉스(검정) 토큰 | 7개 | 보석 토큰 |
| 루비(빨강) 토큰 | 7개 | 보석 토큰 |
| 황금(조커, 노랑) 토큰 | 5개 | 보석이 아님. 와일드카드 |
| 개발 카드 레벨 1 | 40장 | 초록 뒷면 티어 |
| 개발 카드 레벨 2 | 30장 | 노랑 뒷면 티어 |
| 개발 카드 레벨 3 | 20장 | 파랑 뒷면 티어 |
| 귀족 타일 | 10장 | 각 3 명성점 |

- 토큰 총 40개(보석 5색 × 7개 + 황금 5개), 개발 카드 총 90장.
- 개발 카드에는 (a) 명성점(0~5점, 없을 수 있음), (b) 보너스 색 1개(보석 5색 중 하나), (c) 구매 비용(보석 토큰 조합)이 표기된다.
- 귀족 타일에는 (a) 명성점 3점, (b) 방문 조건(특정 색 **보너스**의 개수 요구, 예: 파랑 보너스 3 + 초록 보너스 3 + 하양 보너스 3)이 표기된다.

## 2. 게임 셋업 [룰북]

1. 개발 카드를 레벨별로 **각각 따로** 섞어 3개의 덱을 만들고, 테이블 중앙에 레벨 1(아래)→2→3(위) 순서의 열로 놓는다.
2. **각 레벨에서 4장씩** 공개하여 덱 옆에 나란히 놓는다(총 12장 공개).
3. 귀족 타일 10장을 섞어 **플레이어 수 + 1장**을 공개한다(예: 4인이면 5장). **남은 귀족 타일은 게임에서 완전히 제외**되며 이번 게임에서 사용되지 않는다.
4. 토큰을 색깔별 6개 더미로 정리해 모두의 손이 닿는 곳에 둔다.

### 인원수별 토큰 조정

| 인원 | 각 보석 색 토큰 수 | 황금 토큰 | 공개 귀족 타일 |
|---|---|---|---|
| 2인 | **4개** (각 색에서 3개 제거) | 5개 (변동 없음) | 3장 |
| 3인 | **5개** (각 색에서 2개 제거) | 5개 (변동 없음) | 4장 |
| 4인 | **7개** (제거 없음) | 5개 (변동 없음) | 5장 |

- 룰북 원문: "황금은 건드리지 않는다(Don't touch the gold)." 즉 **황금은 인원수와 무관하게 항상 5개**.
- 그 외 인원수별 차이는 없다(공개 카드 수, 기타 규칙 동일).

### 시작 플레이어
- 가장 어린 플레이어가 선(先)이며, 시계 방향으로 진행한다. [룰북]
- [구현 결정] 온라인 구현에서는 무작위로 선을 정하되, 누가 선인지 명시적으로 저장할 것(게임 종료 판정에 필요, §8 참조).

## 3. 기본 개념

- **토큰**: 보석 토큰 5색 + 황금(조커). 플레이어가 소지한 토큰은 항상 전원에게 공개 정보다. [룰북]
- **보너스**: 구매한 개발 카드 좌상단의 보석 표시. 영구적이며 소모되지 않는다(§5).
- **명성점(프레스티지)**: 구매한 개발 카드의 점수 + 획득한 귀족 타일의 점수(각 3점)의 합. 예약만 한 카드는 점수에 포함되지 않는다.
- **예약 카드**: 손에 보관하는 카드. 최대 3장. 자신만 구매 가능하며, 버릴 수 없다(§4.3).
- 구매한 개발 카드의 보너스·명성점은 항상 전원에게 공개되어야 한다. [룰북]

## 4. 턴 구조 — 4가지 행동 중 정확히 1개 수행 [룰북]

자기 턴에 플레이어는 아래 4가지 행동 중 **반드시 하나만** 선택해 수행해야 한다(원문: "must choose to perform only one of the following four actions"). 행동을 두 개 하거나, 가능함에도 자발적으로 건너뛰는 것(패스)은 불가하다(§9-G 참조).

### 4.1 행동 A — 서로 다른 색 보석 토큰 3개 가져오기
- 서로 다른 색의 보석 토큰을 3개 가져온다.
- **황금 토큰은 이 행동으로 가져올 수 없다** (황금은 보석이 아니며, 예약으로만 획득 가능. §4.3). [룰북]
- [FAQ] 공급처에 서로 다른 색이 3색 미만으로만 남아 있으면, **가능한 만큼(2개 또는 1개)만 가져올 수 있다**. (Dized 공식 FAQ: "the action 'take three different tokens' allows you to pick only two different tokens, or even one, if all the other piles are depleted.")
- [구현 결정] 3색 이상 남아 있을 때 자발적으로 1~2개만 가져오는 것을 허용할지는 룰북에 명시가 없다. **권장: 남아 있는 서로 다른 색 수와 3 중 작은 값만큼 가져오도록 강제**(엄격한 해석). 이 해석은 게임 진행을 막지 않는다 — 10개 제한 초과 시 반납으로 개수 조절이 가능하기 때문(§5).

### 4.2 행동 B — 같은 색 보석 토큰 2개 가져오기
- 같은 색 보석 토큰 2개를 가져온다.
- **조건: 가져오는 시점에 해당 색 토큰이 공급처에 4개 이상 남아 있어야 한다.** 정확히 4개 남은 경우는 가능(가져간 후 2개 남음), 3개 이하 남은 경우는 불가. [룰북 원문: "This action is only possible if there are at least 4 tokens of the chosen color left when the player takes them."]
- [FAQ] 3개 남은 더미에서 2개를 가져올 수 없음이 명시적으로 재확인됨(Dized FAQ: "Can I take two tokens from a pile that has only three tokens left? No").
- 이 행동으로 1개만 가져오는 변형은 없다. 같은 색 1개만 원하면 행동 A로 처리된다(색이 1색만 남은 상황, §9-B).
- 황금 토큰은 이 행동으로도 가져올 수 없다.

### 4.3 행동 C — 개발 카드 1장 예약 + 황금 토큰 1개
- 다음 둘 중 하나를 예약한다:
  - 테이블 중앙에 **공개된** 개발 카드 1장 (아무 레벨), 또는
  - 3개 덱 중 하나의 **맨 위 카드를 비공개로** 뽑아 예약(다른 플레이어에게 보여주지 않음). [룰북]
- 예약과 동시에 **황금(조커) 토큰 1개**를 가져온다.
  - **황금이 공급처에 없으면 황금 없이 예약만 할 수 있다.** [룰북 원문: "If there is no gold left, you can still reserve a card, but you won't get any gold."]
- **예약 카드는 손에 최대 3장.** 이미 3장을 예약 중이면 이 행동을 선택할 수 없다. [룰북]
- 예약 카드는 버릴 수 없으며, 없애는 유일한 방법은 구매하는 것뿐이다. [룰북]
- **예약은 황금 토큰을 얻는 유일한 방법이다.** [룰북]
- 황금을 받아 토큰이 10개를 초과하면 §5의 반납 규칙이 적용된다(방금 받은 황금을 그대로 반납하는 것도 허용).
- 공개 카드를 예약하면 즉시 같은 레벨 덱에서 보충한다(§7). 덱이 빈 레벨은 비공개 예약(덱 예약)이 불가능하다.
- [구현 결정] 정보 공개: 공개 카드를 예약한 경우 그 카드의 정체는 전원이 이미 본 공개 정보로 처리한다. 덱에서 비공개로 예약한 카드는 예약한 플레이어에게만 보이게 한다.

### 4.4 행동 D — 개발 카드 1장 구매
- 다음 둘 중 하나를 구매한다:
  - 테이블 중앙의 공개 카드 1장, 또는
  - **자신이** 이전에 예약해 둔 카드 1장. (다른 플레이어의 예약 카드는 구매 불가.) [룰북]
- 카드에 표기된 비용만큼 토큰을 지불한다. 지불한 토큰(황금 포함)은 공급처로 되돌아간다. [룰북]
- **보너스 할인**: 색별 실지불량 계산(§5.1).
- **황금은 와일드카드**: 황금 1개는 임의의 보석 1개를 대체한다. [룰북]
- 구매한 카드는 자신의 앞에 색별로 정렬해 공개해 둔다(보너스·점수 상시 공개). [룰북]
- 공개 카드를 구매하면 즉시 같은 레벨 덱에서 보충한다(§7).
- [FAQ] 토큰이 충분해 즉시 구매가 가능한 카드도 (구매하지 않고) 예약할 수 있다. 또한 해당 색 보석 토큰을 갖고 있어도 **그 대신 황금으로 지불하는 것은 플레이어의 자유**다(룰북에 금지 규정 없음 — "조커는 어떤 색이든 대체할 수 있다").

#### 4.4.1 지불 판정 알고리즘 (구현용 정식화)
카드 비용을 색 c별 `cost[c]`, 플레이어의 색 c 보너스 수를 `bonus[c]`, 보유 보석 토큰을 `gems[c]`, 보유 황금을 `gold`라 할 때:

1. 색별 실요구량: `need[c] = max(0, cost[c] - bonus[c])`
2. 색별 부족분: `short[c] = max(0, need[c] - gems[c])`
3. **구매 가능 조건**: `Σ short[c] ≤ gold`
4. 지불: 각 색 c에 대해 보석 토큰 `min(need[c], gems[c])`개 이하 + 황금으로 나머지를 충당하되, 색별 지불 합계가 정확히 `need[c]`가 되도록 한다. 플레이어는 (황금이 충분하다면) **어느 색을 황금으로 대체할지 자유롭게 배분**할 수 있다. 지불 총액이 `Σ need[c]`를 넘는 초과 지불은 없다(정확히 필요한 만큼만 지불).
5. 보너스가 비용 전체를 덮으면 **토큰 0개로 무료 구매**가 성립한다. [룰북 원문: "they can even purchase a card without spending any tokens."]
6. 보너스 초과분은 이월·환급되지 않는다(비용은 0 미만이 되지 않음).

## 5. 토큰 10개 소지 제한 [룰북]

- **플레이어는 자기 턴 종료 시점에 토큰을 10개 넘게 가질 수 없다(황금 포함).**
- 초과 시 **즉시** 10개가 될 때까지 반납해야 한다.
- **반납할 토큰은 자유 선택**이며, **방금 가져온 토큰의 전부 또는 일부를 그대로 반납해도 된다.** [룰북 원문: "A player can return all or some of those they've just drawn."]
- 제한은 "턴 종료 시" 기준이므로 턴 도중에는 일시적으로 10개를 초과할 수 있다(예: 10개 소지 상태에서 3개 획득 → 13개 → 3개 반납). 이를 이용해 **토큰 색 교환**이 사실상 가능하다(3개 가져오고 다른 색 3개 반납). [FAQ]
- 구매한 개발 카드(보너스)는 토큰이 아니므로 이 제한과 무관하다. [FAQ]
- 반납된 토큰은 공급처의 해당 색 더미로 되돌아간다.

### 5.1 보너스에 의한 영구 할인 [룰북]
- 이전 턴에 **구매한**(예약이 아닌) 개발 카드의 보너스는 새 카드 구매 시 할인을 제공한다.
- **색별로, 보너스 1개 = 해당 색 토큰 1개와 동등**하다.
- 예: 파랑 보너스 2개 보유 + 비용이 "파랑 2, 초록 1"인 카드 → 초록 토큰 1개만 지불.
- 보너스는 소모되지 않고 **매 구매마다 반복 적용**되는 영구 할인이다.
- 귀족 방문 조건 판정에도 이 보너스(카드 좌상단 보석 표시)만 사용된다 — 토큰은 무관(§6).

## 6. 귀족 방문 [룰북]

- **판정 시점**: 각 플레이어의 **턴 종료 시** 자동 판정한다.
- **조건**: 귀족 타일에 표기된 색별 **보너스**(구매한 카드의 보너스만, 토큰 제외)를 **모두 이상** 보유하면 방문받는다. (원문: "the amount of bonuses (and only bonuses) required")
- **비용 없음**: 귀족 방문에는 토큰도 보너스도 지불하지 않는다. 보너스는 소모되지 않는다.
- **행동이 아님**: 귀족을 받는 것은 행동(action)으로 치지 않는다. 같은 턴에 행동 1개 + 귀족 획득이 모두 일어날 수 있다.
- **거부 불가**: 귀족의 방문은 거부할 수 없다(자동·강제).
- **한 턴에 1장만**: 여러 귀족의 조건을 동시에 충족해도 **한 턴에는 1장만** 받는다. **어느 귀족을 받을지는 해당 플레이어가 선택**한다. 남은 귀족은 타일이 남아 있는 한 다음 자기 턴 종료 시(조건 유지 시) 받을 수 있다.
- 획득한 귀족 타일은 자기 앞에 공개해 두며 **3 명성점**이다.
- **귀족 타일은 보충되지 않는다**(공개 귀족 수는 게임이 진행되며 줄어들기만 한다).
- [구현 결정] 턴 종료 처리 순서 권장: ① 행동 수행 → ② 토큰 10개 제한 검사·반납 → ③ 귀족 방문 판정(복수 충족 시 플레이어 선택) → ④ 게임 종료 트리거 검사(§8). 귀족 판정은 보너스에만 의존하므로 ②와 ③의 순서는 결과에 영향을 주지 않지만, 일관성을 위해 위 순서를 고정한다.

## 7. 공개 카드 보충 [룰북]

- 테이블 중앙의 공개 개발 카드가 **구매되거나 예약되면 즉시** 같은 레벨 덱의 맨 위 카드로 그 자리를 보충한다.
- 게임 내내 각 레벨에는 항상 4장의 공개 카드가 유지되어야 한다. **단, 해당 레벨 덱이 소진된 경우 빈자리는 빈 채로 남는다.** [룰북 원문: "unless the deck in question is empty, in which case the empty spaces also remain empty."]
- 덱에서 비공개 예약(§4.3)한 경우는 공개 카드가 줄지 않으므로 보충이 없다.

## 8. 게임 종료와 승자 판정 [룰북]

1. **트리거**: 어떤 플레이어가 (턴 종료 시점 기준으로) **명성점 15점 이상**에 도달하면 게임 종료가 예약된다. 귀족 점수 포함. 15점 "초과"도 당연히 트리거된다.
2. **마지막 라운드**: 트리거 후 **현재 라운드를 끝까지 진행하여 모든 플레이어가 같은 턴 수를 소화하게 한다.** [룰북 원문: "complete the current round so that each player has played the same number of turns."]
   - 정확한 판정: **선 플레이어(첫 턴을 한 플레이어) 기준**으로, 턴 순서상 마지막 플레이어(선의 오른쪽 이웃)가 그 라운드의 턴을 마치면 게임이 끝난다. 즉, 트리거를 낸 플레이어가 누구든 **선 플레이어의 왼쪽 방향으로 라운드의 잔여 플레이어들이 각 1턴씩** 마저 진행한다. 선 플레이어가 트리거를 내면 나머지 전원이 1턴씩 더 한다. 마지막 순번 플레이어가 트리거를 내면 즉시 종료된다.
   - 마지막 라운드 중에 다른 플레이어가 15점 이상에 도달해도 정상적으로 계산되며, 추가 라운드는 생기지 않는다.
3. **승자**: 라운드 종료 후 **명성점이 가장 높은 플레이어**가 승리한다(귀족 점수를 반드시 합산).
4. **동점 처리**: 동점이면 **구매한 개발 카드 수가 더 적은** 플레이어가 승리한다. [룰북] (예약만 하고 구매하지 않은 카드는 세지 않는다.)
5. [구현 결정] 점수와 카드 수가 모두 같으면 룰북에 규정이 없다. **권장: 공동 승리 처리.**

## 9. 경계 사례(Edge Cases) 총정리

**A. 토큰 부족 시 3개 미만 가져오기** [FAQ]
행동 A(서로 다른 색 3개)에서 공급처에 서로 다른 색이 2색만 남았으면 2개, 1색만 남았으면 1개를 가져올 수 있다. 0개 획득은 행동으로 성립하지 않는다(보석 토큰이 하나도 없으면 행동 A 자체가 불가).

**B. 남은 색이 1~2색뿐일 때** [FAQ]
- 행동 A로 각 색 1개씩(1~2개) 가져온다.
- 그중 어떤 색이 4개 이상 남아 있으면 행동 B(같은 색 2개)도 여전히 가능하다.
- "어느 더미의 마지막 3개 이하 토큰"은 행동 A로만 가져갈 수 있다(행동 B의 4개 이상 조건 때문).

**C. 같은 색 2개의 "4개 이상" 조건** [룰북]
가져오는 시점 기준으로 해당 색이 4개 이상 남아 있어야 한다. 정확히 4개 → 허용. 3개 이하 → 불가. 3개 남은 더미에서 2개 가져오기는 명시적으로 금지 확인됨(Dized FAQ).

**D. 예약 3장 보유 시** [룰북]
행동 C(예약) 선택 불가. 예약 카드는 버릴 수 없고 구매로만 소진된다. 예약 카드가 3장이어도 나머지 행동(A/B/D)은 정상 수행 가능.

**E. 덱 소진 시** [룰북]
- 공개 카드 보충 불가 → 빈자리는 빈 채 유지.
- 그 레벨 덱에서의 비공개 예약 불가. 남은 공개 카드의 예약·구매는 가능.
- 카드가 모자라 15점에 아무도 도달하지 못하는 상황은 이론상 가능하나 룰북 규정이 없다. [구현 결정] 권장: 발생 시(모든 카드 소진 + 진행 불능) 현재 점수로 §8의 3~5항을 적용해 종료.

**F. 황금 토큰의 획득 제한** [룰북]
황금은 행동 A/B로 가져올 수 없다. **예약(행동 C)이 황금을 얻는 유일한 방법.** 황금 더미가 비어 있으면 예약해도 황금을 받지 못한다(예약 자체는 가능).

**G. 패스(행동 생략) 가능 여부** [룰북+FAQ]
- 룰북 원문상 플레이어는 4가지 행동 중 하나를 "반드시(must)" 수행해야 한다. **가능한 행동이 하나라도 있으면 자발적 패스는 불가.**
- 토큰이 10개여도 행동 A/B는 가능하다(가져온 뒤 10개까지 반납하면 됨 — 방금 가져온 것을 그대로 반납해도 됨). 따라서 공급처에 보석 토큰이 1개라도 있으면 막히지 않는다.
- **완전 교착(이론상 가능)**: 공급처에 보석 토큰이 0개(전부 플레이어들 소지) + 예약 3장 + 어떤 카드도 구매 불가인 경우에만 합법 행동이 없다. 룰북은 이 경우를 다루지 않는다. [구현 결정] 권장: 합법 행동 집합이 공집합일 때만 자동 패스(턴 스킵) 처리. 그 외에는 패스 버튼을 제공하지 않는다.

**H. 예약 시 황금으로 10개 초과** [룰북]
10개 소지 상태에서 예약하면 황금을 받아 11개가 된다 → 즉시 아무 토큰이나 1개 반납(방금 받은 황금을 반납해도 됨). 예약 자체는 항상 유효하게 성립한다.

**I. 마지막 라운드 판정의 정확한 기준** [룰북]
"모든 플레이어가 같은 턴 수를 플레이"가 기준. 구현: 매 턴 종료 시 15점 이상자가 존재하는지 검사 → 존재하면 `finalRound = true` → 이후 턴 순서상 선 플레이어 차례가 다시 돌아오기 직전(= 라운드의 마지막 플레이어 턴 종료)에 게임 종료. 트리거 이후 남은 플레이어들의 턴에서 일어나는 득점·귀족 획득도 모두 유효하다.

**J. 귀족 동시 충족** [룰북]
한 턴 종료 시 여러 귀족의 조건을 동시에 충족하면 **플레이어가 1장을 선택**해 받는다. 나머지는 테이블에 남아 이후 턴에서 (조건이 유지되는 한) 획득 가능 — 단 매턴 1장씩. 방문 자체는 거부 불가이므로 "아무 귀족도 안 받기"는 선택지가 아니다.

**K. 즉시 구매 가능한 카드의 예약** [FAQ]
토큰이 충분해도 구매 대신 예약을 선택할 수 있다(견제 목적 등).

**L. 황금 사용의 자유** [FAQ]
해당 색 보석 토큰을 보유하고 있어도 그 자리에 황금을 대신 지불할 수 있다. 지불 배분은 전적으로 플레이어 선택(단 §4.4.1의 총액 규칙 준수).

**M. 지불 토큰의 행선지** [룰북]
구매에 지불된 토큰(황금 포함)은 모두 공급처로 되돌아간다. 다른 플레이어에게 가지 않는다.

**N. 예약 카드의 점수·보너스** 
예약 상태의 카드는 명성점·보너스·귀족 조건·동점 판정용 "구매 카드 수" 어디에도 포함되지 않는다. 구매 완료 시점부터 반영된다. (룰북의 보너스/점수 규정이 모두 "구매(purchased/acquired)한 카드"를 기준으로 하는 데서 도출.)

**O. 토큰·카드 정보 공개** [룰북]
소지 토큰, 구매한 카드의 보너스·점수는 항상 전원 공개. 덱에서 비공개 예약한 카드만 본인 전용 비공개 정보다.

## 10. 턴 처리 의사코드 (구현 참고)

```
turn(player):
  action = choose_one_of_legal_actions(player)   # 합법 행동이 있으면 반드시 선택
  # A: takeDifferent(colors ⊆ 남은 색, |colors| = min(3, 남은 서로다른 색 수), 황금 제외)
  # B: takeSame(color) requires supply[color] >= 4  (황금 제외)
  # C: reserve(card | deckTop) requires player.reserved < 3; if supply.gold > 0: gain 1 gold
  # D: purchase(card ∈ faceUp ∪ player.reserved) requires Σ short[c] ≤ player.gold  (§4.4.1)
  execute(action)
  refill_face_up_slots_if_needed()               # §7 (구매/공개예약 직후 즉시)
  while player.tokenCount > 10:
      player.return_token(free_choice)           # §5
  eligible = nobles.filter(n => ∀c: player.bonus[c] >= n.req[c])
  if eligible: player.gain(player.choose_one(eligible))   # §6, 강제, 1장만
  if player.prestige >= 15: finalRound = true
  if finalRound and player == last_in_round_order: end_game()   # §8
```
